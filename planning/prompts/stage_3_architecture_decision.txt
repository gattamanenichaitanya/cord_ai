# ARCHITECTURAL DECISION PRINCIPLES
1. Simplicity: Use the simplest architecture and configuration that fully satisfies all acceptance criteria in the requirement.
2. Honor Decision Criteria: Adhere strictly to the `decision_criteria` documented in capability entries — these encode expert consultant judgment and technical best practices.
3. Factor in Gotchas: If gotchas are flagged as relevant, explicitly address them in your architectural rationale and parameter design (e.g. choice of re-enrollment toggles, field type constraints, naming immutability).
4. Method Preference: Prefer API-driven execution methods over manual UI steps where both exist, unless UI is explicitly required or API does not support the action.
5. Explicit Parameterization: Be completely explicit in the `parameters` dictionary. Define exact property names, field types, group names, trigger criteria, operators, values, and action configurations.
6. BUNDLE MISSING DEPENDENCIES: If this requirement depends on properties or items that you suspect might be missing, you MUST proactively include their creation operations (e.g., `hubspot.create_custom_property`) in `selected_operations` and include their full configurations in `parameters`. Do not assume they will be created by other requirements; bundle them into this plan so it can execute independently. When generating property creation parameters, the `group_name` MUST be a standard HubSpot group: 'contactinformation', 'companyinformation', or 'dealinformation'. DO NOT invent new group names like 'acs_customer_info'.

BAD: 'Via the HubSpot CRM Properties API v3, create a Number-type custom contact property with internal name acs_risk_score...'
- rationale: A 3-5 sentence explanation justifying why this architecture was selected, citing decision criteria and gotcha mitigation where applicable.
- selected_capabilities: List of capability_names used (e.g. ["workflow", "custom_property"]).
- selected_operations: List of operation_ids that will execute this architecture in approximate execution order (e.g. ["hubspot.create_custom_property", "hubspot.create_workflow"]).

- rejected_alternatives: List of objects detailing alternative approaches considered and why they were rejected (e.g. [{"alternative": "...", "reason": "..."}]).
