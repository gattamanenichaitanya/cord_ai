# GAP CATEGORIES TO DETECT
Analyze the inputs and identify any gaps or risks. Maximum 3 gaps. Prefer 1-2. If you find more, keep only those that will DEFINITELY cause a problem for THIS requirement as written.

CRITICAL RULES FOR GAPS:
- Do NOT flag speculative or future gaps. "If a future workflow does X" is not a gap. Only concrete risks that apply now.
- Do NOT flag Cord AI's own implementation details (API versions, endpoints, execution methods). Those are internal engineering, not consultant concerns.
- A gap is a business or configuration risk stated the way a senior consultant would mention it to a colleague — brief, concrete, actionable.

Categories to consider:

1. State Conflicts & Prerequisites (based purely on Stage 4 findings):
   - DO NOT complain that a 'pre-check' wasn't performed. The Stage 4 Live State Inspection IS the pre-check. Trust its output completely.
   - If Stage 4 found an item already exists but the requirement is trying to CREATE it (e.g., creating a property that is marked "exists": true), flag it as a MEDIUM severity conflict (it will need to be updated or skipped).
   - If a required integration is missing (e.g. Slack not connected in Stage 4) -> HIGH severity, blocks execution.
   - If properties/workflows are missing but will be created by this requirement, flag as LOW severity (this is normal).

2. Live Data Quality Issues (from Stage 4):
   - High null rate on a property used as a trigger source or dynamically referenced (e.g. 22% of contacts have no value for a critical trigger or assignee field) -> MEDIUM severity. Mention how many records will be bypassed.
   - Categorical distribution problems (e.g., all records have the same value in a field meant to drive conditional branching).

3. Graph-Flagged Gotchas & Risks:
   - Match any of the provided product knowledge gotchas to the architecture decision. E.g., if a workflow has re-enrollment enabled and updates properties, check if the `workflow_re_enrollment_behavior` gotcha applies.
   - For property creation, check if `property_internal_name_immutable` gotcha applies.
   - If tracking contact touches, check if `notes_last_contacted_population` gotcha applies.

4. Unresolved Ambiguities:
   - Any missing thresholds, notification preferences, templates, or dynamic routing properties that are not fully specified.

5. Logical/Configuration Risks:
   - Workflows running on all records instead of filtering by lifecycle stage.
   - Filter conditions that can never trigger (e.g., scoring criteria that are mathematically impossible based on current distributions).
